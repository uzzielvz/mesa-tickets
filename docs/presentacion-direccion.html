Objetivo

Plataforma centralizada de operaciones

Un solo lugar donde ir agregando herramientas confiables para el personal, reduciendo trabajo manual y procesos desorganizados.

  • Datos centralizados en lugar de compartir Excel o WhatsApp en cada incidencia.
  • Trazabilidad de quién levantó, quién atendió y cómo se cerró cada caso (auditoría y KPIs).
  • Procesos más rápidos y seguros cuando el volumen o la evidencia justifican dejar el canal informal.
  • Mejor insumo para Data Science: datos estructurados por área, no solo archivos sueltos.
  • Decisiones con datos cuando la información está versionada y consultable.
No implica reemplazar todo de golpe. Cada módulo se valida antes de escalar.

Mantenimiento

Operación de Sistemas hoy

  • Deploys automáticos por git (Vercel, Render).
  • Migraciones de BD versionadas; sin cambios manuales en producción.
  • Stack serverless: sin parches de SO ni servidores que vigilar.
  • Microservicio Free en Render: cold start ~20–30 s tras inactividad (mitigable ~$120 MXN/mes o cron de wake-up).
Carga operativa baja mientras se valida el piloto.
Estado detallado

Listo vs. por ajustar

Listo o en uso

  • Score: producción, equipo comercial.
  • Auth + admin: dominio corporativo, roles, presets (73 personas), catálogo inicial.
  • Tickets (base): formularios dinámicos, 3 tipos sembrados, numeración limpia para primer ticket real.
  • Cartera (piloto): microservicio desplegado (llamada autenticada con token), 5 vistas, RPCs de consulta; carga trazable (quién sube / quién procesa) y fecha de corte automática.
  • IA (piloto): agente con 6 tools; demo en modo mock.

Por ajustar / validar

  • Tickets: entrevistas por área; cola por área; estados; RLS antes de PII masiva.
  • Cartera: auditoría y pruebas por equipo de Data Science; completar ETL; más dashboards solo si hay necesidad validada.
  • IA: conectar datos reales (billing); ampliar tools según prioridad de negocio.
  • Datos: dependencia de Excel Yunius hasta API del RP.
  • Storage: vaciar adjuntos de prueba en bucket de tickets.
No afirmar "todo en producción". DS debe validar métricas de cartera.
Decisiones

Qué necesito de esta junta

  • Confirmar si el enfoque piloto incremental (no big-bang) es el acordado.
  • Priorizar entrevistas / auditoría de procesos para la mesa de tickets.
  • Validar asignación de áreas del personal (presets cargados).
  • Acordar si Data Science revisa métricas y ETL de cartera antes de uso operativo.
  • Aprobar salir de modo mock a datos reales de cartera (billing de Gemini ya activo; costo ~$23–46 MXN/mes).
  • Retomar o cerrar el tema de API Yunius / RP como habilitador de datos.
Salir con prioridades, no con compromiso de construir todo.
